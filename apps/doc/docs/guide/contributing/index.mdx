## The Structural Matrix

The project is a monorepo managed with [pnpm](https://pnpm.io/) and [Turbo](https://turbo.build/).

- `apps/api`: The core registry brain, built with Hono and Cloudflare Workers.
- `apps/cli`: The synaptic interface for managing your registry nodes.
- `apps/doc`: This documentation node, built with Rspress.

## Architectural Foundations

- **Logic Layer**: [Hono](https://hono.dev/) provides the high-performance substrate for our Cloudflare Worker brain.
- **Memory Layer**: [Drizzle ORM](https://orm.drizzle.team/) interacts with [Cloudflare D1](https://developers.cloudflare.com/d1/) for metadata persistence.
- **Artifact Layer**: Tarballs are stored within the [Cloudflare R2](https://developers.cloudflare.com/r2/) object storage mesh.

## Synchronizing Your Local Environment

1. **Clone the Repository**:
   ```bash
   git clone https://github.com/thomas-cogez/npflared.git
   cd npflared
   ```

2. **Establish Dependencies**:
   ```bash
   pnpm install
   ```

3. **Local Manifestation**:
   To initiate the API in development mode:
   ```bash
   pnpm --filter @npflared/api dev
   ```

## Ensuring Systemic Integrity (Testing)

We utilize [Vitest](https://vitest.dev/) and [`@cloudflare/vitest-pool-workers`](https://developers.cloudflare.com/workers/testing/vitest-testing/) to maintain a robust safety net for our core logic.

### Integration Testing (The "Safety Mesh")
Our primary verification method involves stimulating the Worker with real HTTP signals. This ensures:
- Routing resonance (including complex scoped packages).
- Accurate authentication and permission gating.
- Seamless D1 and R2 interactions.

We maintain a **Permission Matrix** (`apps/api/__tests__/integration/permission-matrix.spec.ts`) that verifies the system's access control boundaries against a variety of token and package combinations.

### Running the Verification Suite

- **Full System Check**:
  ```bash
  pnpm run test:single
  ```
- **API Specific Verification**:
  ```bash
  pnpm --filter @npflared/api run test:single
  ```

## Proposing Evolutions (Pull Requests)
1. **Create an Evolutionary Branch**: `git checkout -b feature/my-new-intelligence`
2. **Implement Changes**: Ensure your code resonates with the existing style and includes comprehensive tests.
3. **Systemic Verification**: Run the full test and linting suite.
4. **Signal Your Changes**: Use clear, descriptive commit messages.
5. **Open a PR**: Initiate a pull request against the `main` branch to merge your intelligence into the core.

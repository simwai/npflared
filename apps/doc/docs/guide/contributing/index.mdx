# Contributing

First of all, thank you for considering contributing to Npflared!

## Project Structure

Npflared is a monorepo managed with [pnpm](https://pnpm.io/) and [Turbo](https://turbo.build/).

- `apps/api`: The core registry API built with Hono and Cloudflare Workers.
- `apps/cli`: The Npflared CLI tool.
- `apps/doc`: This documentation site built with Rspress.

## Architecture

- **API**: Built with [Hono](https://hono.dev/) for high performance and compatibility with Cloudflare Workers.
- **Database**: Uses [Drizzle ORM](https://orm.drizzle.team/) with [Cloudflare D1](https://developers.cloudflare.com/d1/).
- **Storage**: Tarballs are stored in [Cloudflare R2](https://developers.cloudflare.com/r2/).

## Development Setup

1. **Clone the repository**:
   ```bash
   git clone https://github.com/thomas-cogez/npflared.git
   cd npflared
   ```

2. **Install dependencies**:
   ```bash
   pnpm install
   ```

3. **Local Development**:
   You can start all applications in development mode from the root:
   ```bash
   pnpm dev
   ```
   This will run the API, CLI (watch mode), and Documentation site simultaneously using Turbo.

   Alternatively, to start only the API:
   ```bash
   pnpm --filter @npflared/api dev
   ```

## Common Scripts

The following scripts are available at the root level:

- `pnpm dev`: Starts all apps in development mode.
- `pnpm build`: Builds all apps.
- `pnpm quality`: Runs all quality checks (linting, type checking, tests).
- `pnpm test:single`: Runs all tests once.
- `pnpm check-format`: Checks code formatting with Biome.
- `pnpm format`: Automatically formats code with Biome.

## Testing Strategy

We leverage [Vitest](https://vitest.dev/) along with [`@cloudflare/vitest-pool-workers`](https://developers.cloudflare.com/workers/testing/vitest-testing/) to provide a robust safety net for our Worker logic.

### Integration Tests (The "Safety Net")

Our primary testing method is hitting the Worker with real HTTP requests using the `SELF` (or `unstable_startWorker`) utility. This ensures that:
- Routing is correct (including scoped packages).
- Authentication and permission gates (like `package:read`, `package:write`) work as expected.
- Database (D1) and Storage (R2) interactions are functional.

We maintain a **Permission Matrix** test suite (`apps/api/__tests__/integration/permission-matrix.spec.ts`) that hammers the Worker with various token and package combinations to prevent regressions in access control.

### Running Tests

- **Run all tests**:
  ```bash
  pnpm run test:single
  ```
- **Run tests for the API only**:
  ```bash
  pnpm --filter @npflared/api run test:single
  ```

## Code Style

We use [Biome](https://biomejs.dev/) for formatting and linting.

```bash
pnpm run check-format
```

## Making a Pull Request

1. **Create a branch**: `git checkout -b my-feature`
2. **Make your changes**: Ensure your code follows the existing style and includes relevant tests.
3. **Verify**: Run tests and linting (`pnpm quality`).
4. **Commit**: Use descriptive commit messages.
5. **Push and PR**: Push your branch and open a PR against the `main` branch.

Thank you again for your help!
